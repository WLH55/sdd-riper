# SDD-RIPER-ONE: Debug Protocol

> 本文件是 `sdd-riper-one` 技能的阶段协议。核心规则见 `references/protocol-core.md`。

设计目标：使用**日志 + Spec + 代码**三角定位策略诊断 bug 或验证功能正确性，不直接修改代码。

**触发**："DEBUG"、"排查"、"日志分析"、"验证功能"

**输入**：
- `log_path`（必填）：日志文件或目录路径
- `issue`（可选）：问题描述、错误消息或堆栈跟踪
- `spec`（可选）：要对照验证的 Spec 文件路径

## 子模式 A：诊断模式（提供 issue 时）

1. **读取日志**：提取错误、异常、堆栈跟踪、关键调用链
2. **加载上下文**：加载相关 Spec 和 CodeMap（如可用）
3. **三角定位**：日志证据 x Spec 预期 x 实际代码逻辑 → 根因候选
4. **输出**结构化诊断报告：
   - 症状：日志显示什么
   - 预期行为（来自 Spec）
   - 根因候选（按优先级，附代码文件/行引用）
   - 建议修复（简要描述，非代码）
5. 需要修复 → 转入 RIPER 流程；简单修复可用 FAST

## 子模式 B：验证模式（提供 spec 但未提供 issue 时）

1. **读取日志**：解析覆盖功能执行路径的全链日志
2. **加载 Spec**：读取验收标准、预期行为和约束
3. **比对**：每个 Spec 标准 → 日志证据 → PASS/FAIL/INCONCLUSIVE
4. **输出**验证报告：
   - 标准 → 日志证据 → PASS/FAIL/INCONCLUSIVE（表格）
   - 整体功能健康评估
   - 缺口：无法从日志验证的标准（建议补充日志或测试）

## 约束

- Debug 模式为只读，不直接修改代码
- 需要代码变更 → 进入 RIPER 或 FAST
- 如有活跃 Spec，结论追加到 Spec 的 Debug Log 章节
- Agent 应利用 CodeMap 索引进行精确的代码导航，而非全仓扫描
